WorkoutSchedule ruleslocked logic decisions
Session generation—
Application generates planned WorkoutSession records from active schedules. Generation runs on app open and looks ahead a configurable window (e.g. 4 weeks)
Status flow—
Generated sessions start as status = planned. User starts → live → completed or skipped. Schedule is independent of session outcome
Template changes—
Changing a template does not retroactively alter already-generated sessions. New sessions pick up the updated template. templateNameSnapshot on session preserves historical display
Deletion—
Deleting a schedule soft-deletes it and stops future generation. Existing planned sessions are not automatically deleted — user clears manually if desired

PR detection ruleslocked logic decisionsupdated
Storage modelappend only
Every PR is a new row. Current PR = best value. Full history preserved for progression charts
1RM formulaEpley
weight x (1 + reps / 30). Applied consistently. Never mix formulas
Set filternormal sets only
warmup, drop, and failure sets excluded from PR detection
Assisted exercisesinverted logic
Lower assistance weight = better. PR detection uses minimum for assisted exercises
Triggeron session complete
PR scan runs when WorkoutSession or CardioSession status moves to completed
Cardio PR distance rulestrict match only
best_pace_at_distance PRs detected only when CardioSession.distanceMiles exactly matches a seeded or user-defined PR distance. No inference from longer sessions. No segment extraction in v1. Prevents questionable PRs from sessions of different distances
Goal checkafter PR write
New PR triggers check against open ExerciseGoals for same exercise + type
user_entered PRsprotected
Auto-detection never overwrites a user_entered PR. New auto_detected PR supersedes by value only if higher
Celebrationdeferred
celebrated = false on write. Application checks on every app open and triggers animation. Sets celebrated = true immediately after
PR types by context—
1rm / max_reps / max_reps_at_weight apply to workout exercises · best_pace_at_distance / longest_distance apply to cardio · max_duration applies to both

Streak calculation ruleslocked logic decisionsupdated
daily—
Streak = consecutive calendar days completed. Breaks on any missed day
specific_days_of_week—
Streak = consecutive scheduled days completed. Unscheduled days ignored
x_per_week—
Streak = consecutive full weeks where timesPerWeek was met. Absolute — no threshold. Never breaks mid-week
x_per_month—
Streak = consecutive full months where timesPerMonth was met. Absolute — no threshold. Never breaks mid-month
Streak unit—
daily/specific_days count completed events · x_per_week counts weeks · x_per_month counts months
Partial first period—
Ignored for x_per_week and x_per_month. Streak starts at second full period
Paused dates—
Excluded from streak calculation. No grace period — user is responsible for pausing honestly
Retroactive pause—
Allowed up to 30 days back. Streak recalculates from HabitLog + HabitStreakPause records at query time. No DB rollback required
Historical eval—
Past completion evaluated against HabitTarget active on that date, not current target
Streak threshold—
For measurable habits only: uses streakCompletionThreshold if set, otherwise full targetValue required. Never applies to frequency-based types

Canonical storage rules — never deviate from these
Medication compliance
MedicationLog is unique on (medicationId, date, doseNumber). totalDosesPerDay on Medication defines how many doses are expected. Compliance = all doseNumber rows for that date are marked taken
Workout scheduling
WorkoutSchedule generates planned WorkoutSession records. Application looks ahead a configurable window on app open. Template changes do not retroactively alter already-generated sessions. templateNameSnapshot preserves display
Snapshot fields
ExerciseLog stores exerciseNameSnapshot, exerciseSetModeSnapshot, exerciseLoadTypeSnapshot, exerciseAttributesSnapshot at log time. WorkoutSession and CardioSession store templateNameSnapshot. Snapshots are write-once — never updated after creation
Polymorphic FK policy
No polymorphic FKs. PersonalRecord uses workoutSessionId and cardioSessionId as separate nullable fields. ExerciseLog.sessionId is required and always references WorkoutSession
Tally periods
TallyLog unique on (tallyItemId, periodStartDate). For daily: periodStartDate = date. For weekly: periodStartDate = week start per UserPreferences.weekStartDay. For never-reset: periodStartDate = TallyItem.createdAt date. Running total = sum of all rows. Current period = row where periodStartDate = current period anchor
Food logging flow
NutritionFoodLogEntry is the audit trail — it records what SavedFood was logged at what quantity. Application multiplies quantity by savedFoodMetricsSnapshot and writes to NutritionLogEntry rows. NutritionLogEntry is the single source of truth for daily totals
Weight
Always stored in lbs. Converted to kg at display layer only
Distance
Always stored in miles. Converted to km at display layer only
Height
Always stored in cm. Converted for display
Duration
Always stored in seconds (SetLog) or minutes (sessions). Never mix within an entity
Pace
Always stored as seconds per mile. Formatted to min/mile or min/km at display layer per UserPreferences.distanceUnit. Never store as formatted string
Dates
Calendar dates (ISO date "2026-04-29") for day anchors. ISO datetime strings for timestamps. All resolved in User.timezone
Age
Never stored. Always derived from UserPreferences.dateOfBirth at query time
Colors
Status colors (complete, missed, pending) are theme tokens only — never stored in DB. User-assigned colors stored as hex strings on Habit, Medication, TallyItem, WorkoutTemplate, CardioTemplate
Seeded data
Seeded records are read-only. Users may add custom records alongside but never modify or delete seeded entries. Identified by isSeeded = true and userId = null
Timestamps
System-generated timestamps (startedAt, completedAt, loggedAt) are immutable after being set. durationMinutes is the user-overridable field. completedAt is never user-editable
Derived values
Never store a value derivable from other stored values without a compelling performance reason. durationMinutes is the exception — directly overridable with durationOverridden flag to track state
Mode switches
AppMode changes are non-destructive. Data surfaces and unsurfaces based on mode. No fresh starts unless user explicitly triggers a reset
Celebrations
PR and goal celebrations use deferred in-app pattern. celebrated = false on write. Application checks on every app open, triggers animation, sets celebrated = true. No push notifications for achievements
Cascading defaults
ExerciseConfig value overrides UserPreferences global default, which overrides hardcoded app fallback. Resolution order: ExerciseConfig.defaultReps/Sets/Rest → UserPreferences.defaultReps/Sets/defaultRestSeconds → app fallback. Applies to reps, sets, and rest timer defaults
Frequency validation
timesPerDay min 2 · timesPerWeek 1–7 · timesPerMonth 1–days in month · timesPerYear 1–365 · intervalDays min 2 · intervalWeeks min 2. Enforced at application layer
Streak thresholds
streakCompletionThreshold applies to measurable habits only. Explicitly ignored for all frequency-based types. x_per_week and x_per_month streaks are absolute — no partial credit
Sport logging
Sports log as CardioSessions with type = sport and free-text sportName field. No separate exercise category for sport
Body metrics storage
BodyMetric is append-only — one row per metric type per date per user. Current value = most recent row for that type. Full history preserved for trend charts. Blood pressure stored as two separate rows: blood_pressure_systolic and blood_pressure_diastolic on the same date. Never derive one from the other
Body metric units
All measurements stored in metric units (cm, %). Converted at display layer per user preference. body_fat_pct stored as a decimal 0–100. resting_heart_rate in bpm. vo2_max in ml/kg/min. blood_pressure values in mmHg. Never store display-formatted strings
Progress photos
ProgressPhoto stores a reference (URI or file path) to the photo — never raw binary data in the DB. Photos stored in device-local or cloud storage. Multiple photos per date are allowed. Deletion is permanent — no soft delete. User owns and controls all photo data
Habit–Tally sync
When Habit.linkedTallyItemId is set, TallyLog.count and HabitLog.value represent the same number and are kept in sync by the application. Tally is the primary input surface — writes to TallyLog propagate to HabitLog.value, and writes to HabitLog.value propagate to TallyLog.count. Only valid for measurable habits (habitType = 'measurable'). App enforces unit consistency between the linked Habit and TallyItem at creation time. Linking a binary habit to a TallyItem is not permitted
Day boundary
UserPreferences.dayEndTime defines when the user's day ends. Default "00:00" (midnight), configurable up to "04:00". Activity logged between midnight and dayEndTime counts toward the previous calendar date. Storage always uses the actual calendar date — the offset is applied only at the display and streak calculation layer. No schema changes required on log entities
Progression intent
ExerciseLog.progressionIntent is user-declared at session end, per exercise. It is optional and never system-derived. The workout player queries the most recent completed ExerciseLog for a given exerciseId (normal sets only) at session start and surfaces progressionIntent alongside the last session summary (e.g. "3 sets · 140/140/130 lb · May 1 ↑"). Null = user did not log a signal for that exercise that session
Retroactive window
All retroactive entry is capped at 3 calendar days from the end of the current day (dayEndTime). For WorkoutSession and CardioSession, the window is evaluated against workoutDate. For HabitLog, MedicationLog, and DailyLog, the window is evaluated against date. Enforced at the application layer. Dates outside the window are not selectable in any date picker — an inline explanation is shown to the user
Session completedAt timing
WorkoutSession.completedAt is set when the user taps Save on the session summary modal, not when End Session is tapped. End Session opens the summary but does not write completedAt or change status to completed. The session remains status = live until Save fires on the summary
